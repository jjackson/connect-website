Program file · Reading Glasses · April 2026

Readers — can the
Connect model do
vision as well as it
does child health?

Readers is the first non-child-health deployment of the Connect LDVP architecture. In partnership with Restoring Vision, FLWs screen adults for presbyopia and distribute reading glasses. The clinical domain is different. The infrastructure is the same — Trial Run contracting, Learn-app training, verified visits, performance-based payment. This file is less about evidence (we are early) and more about the question the program was designed to answer.

Program file · Scope: Readers. Evidence base is developing; claims named as such.
The question · 1

Is Connect's architecture genuinely sector-agnostic, or is it a child-health platform that looks generic?

CHC proved the Connect architecture works for childhood nutrition and vaccination. KMC proved it works for newborn care. ECD proved it works for behavior-change-heavy interventions. The open question — and it's a real one — is whether the architecture generalizes to sectors that don't live on Dimagi's home turf. Presbyopia screening is a different clinical domain. The screening protocol is optical. The "delivery" is a commodity (reading glasses). The verification elements are different — a photo of glasses fitted, not a MUAC tape reading. Readers is where we find out if the platform actually generalizes, or if our confidence was mostly the result of staying close to what we already knew.

The question · 2

What's the same as the rest of Connect?

Readers inherits the Connect LDVP architecture without structural modification. Learn runs the same 5-layer training model: Learn app content (optical screening, presbyopia basics, glasses fitting), knowledge test, AI coach with Readers-specific scenarios, peer practice, supervised observation. Deliver contracts LLOs under the Trial Run model — the exact same structure that scaled CHC to 44 LLOs. Verify runs the four-layer stack, with the algorithmic layer being retrained on Readers-specific data patterns. Pay is performance-based, per verified visit. There is no Readers-specific platform surface. That's the point.

Inherits from the platform

  • Trial Run contracting
  • 5-layer Learn sequence
  • Four-layer verification
  • Performance-based payment
The question · 3

What's different?

Child health work is predominantly measurement + counseling + referral. Readers is predominantly screening + distribution. That changes several things at the visit level. The visit is shorter. The success criterion is tighter — did the right adult receive the right pair of glasses. The fraud surface is different; swapping a pair of glasses isn't the same attack as falsifying a MUAC reading. The Deliver-app UI has been adapted accordingly. But — and this is what we're watching — the platform surfaces haven't needed to change. Only the content and the configuration did. If that holds across Readers' early campaigns, it's evidence that the sector-agnostic claim is real.

Different visit shape

  • Screening + distribution (not counseling)
  • Commodity delivery
  • Different fraud attack surface
The question · 4

What will we know in 12 months?

Readers is early. We don't yet have the scale-period cost curve, the fraud-model AUC on Readers-specific data, or the long-tail LLO performance distribution. We'll publish each of those as the evidence base develops. What we already have is a working Trial Run portfolio, a functioning Learn-app training sequence, and a verification stack that has absorbed a new sector without architectural change. The missing pieces are exactly the ones that only scale will produce.

If you're a funder considering vision-loss programs or a partner working in primary-care screening, this is the moment to join. The Connect architecture is proven in child health and actively testing in Readers. Early partners shape what "vision on Connect" becomes.
